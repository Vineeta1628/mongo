📚 Learning Goals (Week 1-3)

1. Installation & Setup

2 hours
  • Install MongoDB Community Edition
  • Install MongoDB Compass (GUI tool)
  • Understand MongoDB vs MySQL architecture
  • Learn what databases, collections, and documents are

Install MongoDB, create a database called "my_first_db", and add 3 documents using Compass.

2. CRUD Operations (The Foundation)

1 week

CRUD = Create, Read, Update, Delete - The bread and butter of database work!

Create: insertOne(), insertMany()
Read: find(), findOne()
Update: updateOne(), updateMany()
Delete: deleteOne(), deleteMany()
🎯 Mini Project:

Build a "Todo List" app - Add tasks, mark complete, update text, delete tasks.

3. Basic Queries & Filters

3 days
  • Query operators: $eq, $gt, $lt, $in
  • Logical operators: $and, $or, $not
  • Sorting and limiting results
  • Understanding query performance basics
Example: Find all users over 25 years old
db.users.find({ 
  age: { $gt: 25 } 
}).sort({ 
  age: 1 
}).limit(10);

📚 Learning Goals (Week 4-8)

1. Indexing & Performance

1 week

The Library Analogy: Imagine finding a book in a library with no catalog (no index) - you'd check EVERY shelf! With a catalog (index), you go straight to the right shelf. That's what indexes do for databases! 📚

  • Single field indexes
  • Compound indexes (multiple fields)
  • Understanding explain() for query analysis
  • When NOT to use indexes (write-heavy collections)
Create index on email field
// Before: Query takes 2 seconds
db.users.find({ email: "alex@example.com" })

// Create index
db.users.createIndex({ email: 1 })

// After: Query takes 0.002 seconds!
// 1000x FASTER! 🚀
🎯 Mini Project:

Add 100,000 documents to a collection. Measure query time WITHOUT index. Add index. Measure again. See the magic! ✨

2. Aggregation Pipeline (Your Superpower!)

2 weeks

The Assembly Line Analogy: Think of data flowing through an assembly line. Each stage transforms it: filter → sort → group → calculate. That's aggregation! 🏭

$match: Filter documents (like WHERE in SQL)
$group: Group and calculate (like GROUP BY)
$sort: Sort results
$project: Select/transform fields
Real Example: Calculate total sales by category
db.orders.aggregate([
  // Stage 1: Filter completed orders
  { $match: { status: "completed" } },
  
  // Stage 2: Group by category and sum
  { $group: {
      _id: "$category",
      totalSales: { $sum: "$amount" },
      orderCount: { $sum: 1 }
  }},
  
  // Stage 3: Sort by total sales
  { $sort: { totalSales: -1 } }
]);
🎯 Mini Project:

Build an analytics dashboard - total revenue, top products, sales by month, average order value.

3. Data Modeling (The Art of Design)

1 week

The million-dollar question: To embed or to reference? 🤔

📦 Embedded (Nested)
user: {
  name: "Alex",
  address: {
    street: "...",
    city: "..."
  }
}

Use when: Data read together, 1-to-1 or 1-to-few relationships

🔗 Referenced (Separate)
user: {
  name: "Alex",
  ordersIds: [
    123, 456, 789
  ]
}

Use when: Many-to-many, data updated separately, large arrays

🎯 Mini Project:

Design a blog schema - posts, comments, authors, tags. Decide what to embed vs reference. Justify your choices!

📚 Learning Goals (Week 9-12)

1. Replication & High Availability

1 week

The Backup Band Analogy: You're watching a concert. Lead singer gets sick. Backup singer steps up seamlessly. Show goes on! That's replica sets - automatic failover! 🎤🎸

🎯
Primary

Handles all writes

📋
Secondary

Replicates data

📋
Secondary

Replicates data

Primary fails? → Secondary promoted automatically! ✅

  • Setting up replica sets (minimum 3 nodes)
  • Understanding election process
  • Read preferences (primary, secondary, nearest)
  • Handling network partitions

2. Sharding (Horizontal Scaling)

1 week

The Library Network Analogy: One library can't hold all books in the world. So cities build multiple libraries, each holding different sections. That's sharding - split data across servers! 📚📚📚

How Data Gets Distributed
Shard 1: Users A-H (3.3M docs)
Shard 2: Users I-P (3.3M docs)
Shard 3: Users Q-Z (3.4M docs)
Total: 10M users distributed evenly! ⚖️
  • Choosing the right shard key (critical decision!)
  • Setting up sharded clusters
  • Understanding chunk migration
  • Query routing with mongos

3. Security & ACID Transactions

1 week
🔒 Security Essentials
  • Authentication (users & passwords)
  • Authorization (roles & permissions)
  • Encryption at rest
  • TLS/SSL connections
💾 Transactions
  • Multi-document ACID transactions
  • Session management
  • Rollback on errors
  • When to use transactions
Example: Bank transfer with transaction
const session = client.startSession();
session.startTransaction();

try {
  // Debit from Account A
  await accounts.updateOne(
    { _id: "accountA" },
    { $inc: { balance: -100 } },
    { session }
  );
  
  // Credit to Account B
  await accounts.updateOne(
    { _id: "accountB" },
    { $inc: { balance: 100 } },
    { session }
  );
  
  await session.commitTransaction();
  console.log("Transfer successful! ✅");
} catch (error) {
  await session.abortTransaction();
  console.log("Transfer failed, rolled back! ❌");
}

❓ Common Interview Questions

Q1 How long does it realistically take to learn MongoDB from scratch? ▼

Answer:

It depends on your goals and prior experience:

  • Basic Proficiency (3-4 weeks): You can perform CRUD operations, write queries, and build simple apps. Good enough for junior roles or small projects.
  • Job-Ready Skills (8-12 weeks): You understand indexing, aggregations, data modeling, and can contribute to production codebases. Suitable for mid-level positions.
  • Production Expertise (6-12 months): You can design scalable systems, handle replication/sharding, optimize performance, and mentor others. Senior-level capability.

Key Factors:

  • Prior database experience (SQL background helps)
  • Programming knowledge (JavaScript/Python recommended)
  • Daily practice time (1-2 hours/day = faster progress)
  • Hands-on projects (crucial for retention)

Realistic Timeline: Most developers become productive in 2-3 months with consistent practice.

Q2 Should I learn SQL before MongoDB, or can I start with MongoDB directly? ▼

Answer:

You can start with MongoDB directly! Here's why:

Reasons MongoDB is beginner-friendly:

  • No schema anxiety: You don't need to design perfect tables upfront
  • Natural data structure: JSON documents match how you think in code
  • Faster gratification: Build working apps quicker
  • Modern ecosystem: Great tools, documentation, and community

When SQL knowledge helps:

  • Understanding database concepts (indexes, transactions, normalization)
  • Translating SQL queries to MongoDB equivalents
  • Making architectural decisions (when to use SQL vs MongoDB)

Recommendation: Start with MongoDB if you're building modern web apps. Learn SQL later when you need it for specific projects or job requirements. Many successful developers learned MongoDB first!

Q3 What's the most important skill to master in MongoDB? ▼

Answer:

If I had to pick ONE skill that makes the biggest impact: Data Modeling

Why Data Modeling matters most:

  • Makes or breaks performance: Bad schema = slow queries, no amount of indexing can fix it
  • Affects entire application: Your data model shapes your code architecture
  • Hard to change later: Unlike SQL, there's no easy "ALTER TABLE" equivalent
  • Separates juniors from seniors: Anyone can learn syntax; modeling requires experience

The critical decision: Embed vs Reference

  • Embed when: Data read together, 1-to-few relationships, bounded arrays
  • Reference when: Many-to-many, independent updates, unbounded growth

Example: E-commerce orders:

  • ✅ Embed: Order items (always read together)
  • ❌ Don't embed: Customer details (updated independently)

How to master it: Build projects, make mistakes, refactor, repeat. Experience is the only teacher!
